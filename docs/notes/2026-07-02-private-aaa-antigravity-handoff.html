FALO IM Watch / LINE Lab - aaa Antigravity 交接說明

本文件給 Force 與 aaa / Antigravity 參考開發用。此版本保留內部參數、架構判斷與產品思路；對外分享前需另做去機敏版本。

1. 目前專案定位

FALO IM Watch 的核心不是單純做聊天機器人，而是把高頻訊息工作場中的 LINE、Telegram、FALO IM 等輸入來源，整理成可被 AI 協作、任務追蹤、風險偵測與知識管理使用的資料層。

LINE 是重要資料來源之一，但不是唯一來源。LINE 更接近不得不接的現場資料搜集器；真正的指揮調度與 AI cowork 可以放到 Telegram、FALO 自建 IM、後台戰情中心或其他更彈性的介面。

3. 地端版：Local AI Node

地端版是本機 AI node 與戰情中心雛形，可用 Python、Win/Mac、ngrok 與本地儲存先跑出高價值功能。未來 GAS、local webhook、匯入紀錄與 Cloudflare Gateway 都要盡量維持資料格式相容。

LINE webhook / ngrok
  -> local Python server
  -> events.jsonl
  -> per-chat folder
  -> media files
  -> local admin dashboard
  -> AI Parser / task extraction / KM

4. LINE 對話紀錄匯入 + 官方助手整合 KM

LINE 官方帳號 webhook 只能接到 Bot 在場且 webhook 啟用後的新事件。若要補歷史資料、指定群組過往討論，或沒有 Bot 在場的既有資料，就需要支援 LINE 對話紀錄匯入。

入口用途資料特性
LINE Official Account webhook收新訊息、正式可維護結構化、事件即時、受 LINE 官方限制
本機 OCR / Local observation看使用者原本正在看的畫面適合桌面輔助與特殊情境
LINE 對話紀錄匯入補歷史資料與指定群組長期知識批次匯入、可重跑、適合 KM

建議使用 GitHub 上已有的 LINE 訊息解析套件或社群 parser 作為起點，先研究 LINE app 匯出的文字紀錄格式、不同平台與語系差異，以及圖片、貼圖、檔案、語音訊息在匯出檔中的表示方式。

source_type = webhook | gas | local | line_export | manual_import

5. 共同資料欄位

不管資料來自 GAS、地端 webhook、OCR 或 LINE export，都要轉成同一個事件模型，至少包含 captured_at、line_timestamp、source_type、bot_basic_id、source_key、user_id、group_id、message_id、message_type、text、media_status、media_drive_url 與 raw_json。

6. KM 與 AI cowork

原始訊息
  -> 標準化事件
  -> 對話片段 / thread
  -> 任務、決策、風險、客戶需求、待追蹤事項
  -> 可搜尋 KM
  -> AI cowork / 戰情中心 / 提醒與分派

LINE 官方助手負責接新事件，指定群組對話匯入負責補歷史脈絡，AI Commander 負責把訊息變成任務、風險、決策與知識。

7. Cloudflare 後續主軸

後續雲端版可能改以 Cloudflare 作為主軸，GAS 保留為雙版開發。GAS 適合 MVP、Google 生態系與教材；Cloudflare 適合正式 SaaS gateway、高頻、多租戶與多 webhook。

Cloudflare 版可考慮 Workers 接 webhook、Queues 排隊、D1 做事件索引、R2 存媒體、Pages / Zero Trust 做管理後台。

8. aaa 接手優先順序

  1. 保留並驗證目前 GAS 版 webhook。
  2. 補強 GAS README、HTML 監看頁與 proxy 狀態顯示。
  3. 做 LINE export import adapter，先只支援文字。
  4. 將匯入事件寫入同一套 normalized schema。
  5. 地端 AI Commander 支援來源切換：local / GAS / import。
  6. 補 Cloudflare gateway 的最小 PoC，不急著取代 GAS。

9. 不建議做的事

  • 不要以 LINE 非官方 userbot / 逆向協議作為正式路線。
  • 不要把 NotebookLM 當 source-of-truth。
  • 不要把 webhook 新資料與匯入歷史資料混在一起而不標示來源。
  • 不要在 webhook 主流程中做長時間媒體下載或 AI 分析。
